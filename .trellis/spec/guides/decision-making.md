# 技术决策方法论

## 决策原则

### 1. YAGNI (You Aren't Gonna Need It)

不要实现当前不需要的功能:

```
❌ 错误: "以后可能用到，先加上这个配置项"
✅ 正确: 等真正需要时再加
```

### 2. 最小变更

选择改动最小的方案:

```
方案A: 修改1个文件，加5行代码
方案B: 新建3个文件，重构2个模块
→ 选择方案A
```

### 3. 渐进式优化

先让它工作，再让它更好:

```
第一步: 简单实现，满足需求
第二步: 如果有性能问题，再优化
第三步: 如果还不够，引入更复杂的方案
```

## 决策流程

```
1. 明确问题
   ↓
2. 收集信息
   ↓
3. 列出方案
   ↓
4. 评估利弊
   ↓
5. 选择方案
   ↓
6. 实施验证
```

## 常见决策场景

### 1. 是否引入新依赖

考虑因素:
- 是否有现成的内部实现?
- 依赖的维护状态如何?
- 引入的复杂度 vs 收益
- 是否有更轻量的替代?

```
需要JSON处理:
- 简单场景 → 使用 Jackson (已引入)
- 复杂场景 → 考虑 Gson 或其他
```

### 2. 是否使用缓存

决策树:
```
数据是否会被频繁访问?
├─ 是 → 数据变更频率如何?
│       ├─ 低 → 使用长时间缓存
│       └─ 高 → 使用短时间缓存或不用缓存
└─ 否 → 不使用缓存
```

### 3. 同步 vs 异步

```
需要立即返回结果?
├─ 是 → 同步处理
└─ 否 → 操作耗时?
        ├─ 短 (<100ms) → 同步处理
        └─ 长 (>100ms) → 异步处理
```

### 4. 数据库设计

```
需要事务?
├─ 是 → 保持关联在同一个数据库
└─ 否 → 考虑是否需要分库分表

关联数据大小?
├─ 一对一 → 外键或嵌入
├─ 一对多 → 外键
└─ 多对多 → 关联表
```

## 技术选型考量

### 稳定性优先

```
成熟框架 > 新框架
长期支持版本 > 最新版本
广泛使用 > 小众使用
```

### 社区活跃度

检查:
- GitHub Stars 和更新频率
- Issue 响应时间
- 文档完善程度
- 社区讨论活跃度

## 决策记录

重要决策应记录在文档中:

```markdown
## 决策: 使用 Redis 作为分布式缓存

### 背景
系统需要支持多实例部署，本地缓存无法共享

### 方案对比
| 方案 | 优点 | 缺点 |
|------|------|------|
| Redis | 功能丰富，生态完善 | 需要额外部署 |
| Memcached | 简单高效 | 功能较少 |
| 本地缓存 | 无网络开销 | 无法共享 |

### 决策
选择 Redis

### 原因
1. 功能满足需求
2. 团队有使用经验
3. 社区活跃，文档完善
```
